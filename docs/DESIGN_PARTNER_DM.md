# 設計パートナー向けDM（3パターン）

## 🎯 使い分け

| パターン | 対象 | 刺さるポイント |
|---------|------|--------------|
| **A** | AI/エージェント本番運用チーム | 事故が怖い、承認フロー必須 |
| **B** | WebhookまみれSaaS | 配送と運用が地獄 |
| **C** | 監査/コンプラが強い領域 | 説明責任が必須 |

## 📧 パターンA: AI/エージェント本番運用チーム向け

### **件名**
```
AI/自動化の「本番実行」を安全にする Confirm Layer（45秒デモ）
```

### **本文**
```
いまエージェント/自動化を本番で回すときに詰まるのって、
「実行前承認」「監査ログ」「冪等」「事故時に止める」だと思ってます。

Yohakuは /plan→/approve→/confirm の"確定レイヤー"をAPIで提供して、
- KYA（誰が/誰の代理で実行したか）を必ず記録
- Ledger（append-only）で監査に耐える
- Freezeで即停止
- Webhook配送は署名+timestamp+retryで確実に届く
を「30分導入」で通します。

もしすでに本番運用してるなら、15分だけ現状の運用フロー見せてください。

（設計パートナー枠：60日無料・要週次フィードバック）

質問:
いま "人間承認" を入れたいアクション（通知/チケット/権限/デプロイ等）あります？
```

### **刺さる理由**
- 「事故が怖い」を直接解決
- 「承認フロー」が最初から入ってる

## 📧 パターンB: WebhookまみれSaaS向け

### **件名**
```
Webhook運用（署名/リトライ/冪等/監査）を30分で外出しできます
```

### **本文**
```
Webhookって作るのは簡単なんですが、
本番で困るのが「署名」「リトライ」「冪等」「失敗原因追跡」「止める」あたりで、
気づくと運用コストが増え続けます。

Yohakuは "Webhook Delivery" をやりつつ、
AI/自動化の時代に必要な「承認」「証跡」「停止」を最初から持った Confirm Layer です。

- Approve（TTL10分）で実行前に止められる
- Receiptで誰が何を確定したかが一発で説明できる
- Jobsで配送状態が見える/再試行できる
- Freezeで事故った瞬間に止められる

45秒デモ: [URL]
スクショ: [URL]

「自前Webhook運用」をやってるなら、置き換え or 併用の設計を一緒に15分で引きます。

質問:
いまWebhookの"障害対応/再送/冪等/監査"って誰がオンコールですか？
```

### **刺さる理由**
- Webhook運用の地獄を直接解決
- Svix/Hookdeckとの差別化（承認 + 証跡 + 停止）
- 「運用コスト」を削減

## 📧 パターンC: 監査/コンプラが強い領域向け

### **件名**
```
AI/自動化の「実行責任」を監査に耐える形で残す（KYA + Ledger + Freeze）
```

### **本文**
```
AI/自動化を本番で回すほど、
「誰が承認したか」「誰の代理で実行したか」「止められるか」が後から必ず問われます。

Yohakuは、実行前承認（Approve）→ 実行確定（Confirm）→ 証跡（Receipt/Ledger）を
最小構成で提供します。
- KYA（executor/principal）を必ず記録
- append-only ledgerで監査可能
- Freezeで即停止（理由ログ込み）

45秒デモ: [URL]
スクショ: [URL]

もし "実行責任" の設計で詰まっている案件があれば、設計パートナー枠で一緒に作り切れます。

質問:
監査/稟議で「誰が承認した？」を問われたこと、ありますか？
```

### **刺さる理由**
- 「実行責任」を直接解決
- KYA + Ledgerで監査に耐える
- SOC2 / GDPR / EU AI Act対応

## 🎯 DM送信後のアクション

### **1. 15分ヒアリング**
- 現状の運用フロー確認
- 「どこが危ないか」を特定
- Yohakuで潰せるかを判断

### **2. PoC設計（30分）**
- 設計パートナー枠（60日無料）
- 週次フィードバック
- 一緒に作り切る

### **3. 成功条件**
- 週次confirmが増える
- 「これ無しで本番は無理」を言わせる
- LOI or paid pilot

## 📊 狙う3社の型

### **① AI Agentを本番で動かしてるチーム**
- **ペイン**: 事故が怖い（勝手に実行、二重実行、監査不能）
- **刺さる理由**: Approve + KYA + Ledger + Freezeで責任が成立する
- **例**: OpenAI Assistants、LangChain利用企業

### **② Webhook連携が多いSaaS**
- **ペイン**: 配送と運用が地獄（署名・Retry・冪等を自前実装）
- **刺さる理由**: 30分導入 vs 3週間自前実装
- **例**: CRM、マーケティングオートメーション

### **③ 監査/コンプラが強い領域の自動化チーム**
- **ペイン**: 説明責任が必須（SOC2、GDPR、EU AI Act）
- **刺さる理由**: Ledger（append-only）+ KYAで監査に耐える
- **例**: 金融、製造業のAI部門
